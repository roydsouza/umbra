# MissionControl: Security Posture & Threat Mitigation

This document outlines the security strategy for the MissionControl Onion Service, focusing on proactive defense against common attacks and preventing accidental information leakage.

## 1. Core Security Principles

*   **Minimalism:** The service should have the absolute minimum number of dependencies and exposed features necessary to function. Every dependency is a potential attack vector.
*   **Principle of Least Privilege:** The service should run with the lowest possible system privileges. It should only have read access to the data it needs to serve and write access only to its designated log file.
*   **Defense in Depth:** We will layer multiple security controls, from the code level to the operational environment.

## 2. Threat Mitigation Strategies

### A. Intrusion Protection & Code Hardening

*   **Memory Safety:** The use of **Rust** is the primary defense against memory-related vulnerabilities (e.g., buffer overflows, use-after-free) common in C/C++ based services.
*   **Dependency Auditing:**
    *   **`cargo-audit`:** Integrate into the CI/CD pipeline (or pre-commit hooks) to continuously scan for known vulnerabilities in dependencies.
    *   **`cargo-deny`:** Configure to enforce policies on dependency licenses, block duplicate dependencies, and disallow known insecure crates.
*   **Static Analysis:** Use `clippy` with its most restrictive settings (`-D clippy::all -D clippy::pedantic`) to enforce high-quality, secure coding patterns.

### B. Traffic Analysis & Fingerprinting Defense

*   **Vanguards:** The `arti-client` should be configured to use Vanguards. This feature adds an additional long-lived guard node to the Tor circuit, making it significantly harder for an attacker to perform traffic correlation attacks to de-anonymize the service. *Research required: Confirm Vanguards support and configuration in the `arti-client` crate.*
*   **Header Stripping:** As outlined in `ARCHITECTURE.md`, an Axum middleware will be used to strip identifying headers like `Server`, `Date`, and `X-Powered-By`. This prevents service and host fingerprinting.
*   **Uniform Response Sizes:** Where feasible, consider padding responses to a uniform size to make traffic analysis based on response length more difficult. This is an advanced technique and may have performance implications.
*   **Stateless Design:** The service should be designed to be as stateless as possible. This reduces the attack surface and makes it harder for an attacker to identify unique user sessions through side-channels.

### C. Protection Against Denial-of-Service (DoS)

*   **Client Authentication:** For sensitive or high-resource endpoints, consider implementing Tor's Onion Service Client Authentication. This would require authorized users to possess a specific key or token to access the service, effectively making it a private Onion Service.
*   **Proof-of-Work:** As a more advanced defense, the service could require clients to solve a small computational puzzle (Proof-of-Work) before processing a request. This can deter automated, high-volume attacks.

## 3. "Deal Breakers": Preventing Secret Leakage

This section details artifacts that **MUST NEVER** be committed to the Git repository. Accidental commitment of these secrets would compromise the entire security and anonymity of the service.

*   **Onion Service Private Keys:**
    *   **Files:** `hs_ed25519_secret_key`, `hs_ed25519_public_key`, etc.
    *   **Threat:** Leaking the private key allows anyone to impersonate your Onion Service.
    *   **Protection:** These keys will be generated by Arti and stored in a dedicated, git-ignored `keys/` directory within the `umbra/missioncontrol` project.

*   **Hardcoded C-Tor Ports or Paths:**
    *   **Content:** Any code that hardcodes the C-Tor SOCKS port (`9050`) or paths related to the C-Tor installation (`/usr/local/tor-m5`).
    *   **Threat:** Accidentally using the legacy C-Tor instance would bypass the security and memory-safety guarantees of Arti. It introduces a dependency that this project is explicitly designed to avoid.
    *   **Protection:** All Tor-related interaction must be done programmatically through the `arti-client` crate. Configuration for Arti (such as its SOCKS port `9150`) should be loaded from the `umbra` project's configuration, not hardcoded.

*   **Sensitive Configuration:**
    *   **Files:** Any `.env` files or configuration files containing passwords, API keys, or other credentials.
    *   **Threat:** Leaking credentials could compromise other parts of the EventHorizon ecosystem.
    *   **Protection:** Use a `.env` file for local development (and add it to `.gitignore`). For production, inject secrets via environment variables or a secure vault, not via version-controlled files.

*   **Personal Identifiable Information (PII):**
    *   **Content:** Usernames, email addresses, specific location data, or anything that can be traced back to the user's real-world identity.
    *   **Threat:** Defeats the purpose of an anonymous service.
    *   **Protection:** Perform regular audits of all code and documentation to ensure no PII is hardcoded. Use placeholder or generic information in all version-controlled files.
